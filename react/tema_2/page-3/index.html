    3 Efectos¶

    Introducción¶

    En React, los efectos son acciones que ocurren fuera del flujo de renderizado normal y que generalmente están relacionadas con la interacción con el DOM, llamadas a servicios web, suscripciones a eventos o cualquier otra operación asíncrona.

    En versiones anteriores de React, para manejar efectos secundarios, se utilizaban principalmente los métodos de ciclo de vida de los componentes de clase, como componentDidMount y componentDidUpdate. Sin embargo, con la introducción de React Hooks en React 16.8, la gestión de efectos secundarios se ha vuelto más sencilla y flexible.

    En el contexto de React Hooks, el hook useEffect se utiliza para manejar efectos secundarios en componentes funcionales.

    useEffect¶

    El useEffect es un hook que nos permite ejecutar código arbitrario cuando el componente se monta en el DOM y cada vez que se actualiza las dependencias que le indiquemos. El useEffect es un método que recibe dos parámetros: un callback que se ejecutará cuando se monta el DOM, ya sea al principio o a cambiar el estado de alguna de las dependencias (este parámetro es obligatorio); y una lista de dependencias de las que dependerá el useEffect, que en caso de que cambien de estado producirá una nueva llamada del callback:

    useEffect(callback, [dependencias])

    El useEffect se ejecuta mínimo una vez, cuando el componente se monta por primera vez en el DOM.

    useEffect(() => {
    console.log('Probando use effect')
})

const App = () => {
    return (
        <h1>Prueba del efecto</h1>
    )
}

    Dependencias¶

    Las dependencias del useEffect son las variables de las que depende el efecto en sí, es decir, las variables que si cambian de valor producirán que el callback del efecto se vuelva a ejecutar.

    const App = () => {
    const [state, setState] = useState(false)

    useEffect(() => {
        console.log('Ejecutando')    
    }, [state])

    return (
        <>
            <h1>Ejemplo de useEffect</h1>
            <button onClick={() => setState(!state)}>Cambiar estado</button>
        </>
    )
}

    En el ejemplo anterior se mostrará el console.log cada vez que cambie el estado de la variable state. La primera ejecución se hace cuando se monta el componente en el DOM.

    En caso de que no se indiquen dependencias, el efecto será ejecutado cada vez que se renderice el componente, lo que a veces puede ser contraproducente para nuestro proyecto:

    const App = () => {
    const [state, setState] = useState(false)

    useEffect(() => {
        console.log('Ejecutando')    
    })

    return (
        <>
            <h1>Ejemplo de useEffect</h1>
            <button onClick={() => setState(!state)}>Cambiar estado</button>
        </>
    )
}

    Para ejecutar el efecto una única vez y evitar que se realice en cada renderizado sin que dependa de ningún estado, es conveniente que la lista de las dependencias este vacío.

    const App = () => {
    const [state, setState] = useState(false)

    useEffect(() => {
        console.log('Ejecutando')    
    }, [])

    return (
        <>
            <h1>Ejemplo de useEffect</h1>
            <button onClick={() => setState(!state)}>Cambiar estado</button>
        </>
    )
}

    Proyecto MouseEffect¶

    En este proyecto vamos a crear un componente que muestre una bola que persiga al puntero por toda la pantalla. En nuestro componente principal vamos a introducir el componente dentro de un main:

    function App() {
  return (
    <main>
      <FollowMouse />
    </main>
  )
}

export default App

    Controlamos si se desea seguir o no el puntero con un estado que se cambiará al hacer clic en el botón (el mensaje del botón también cambiará):

    const FollowMouse = () => {

    const [enabled, setEnabled] = useState(false)

    return (
        <>
            {/*...*/}
            <button onClick={() => setEnabled(!enabled)}>{enabled ? 'Desactivar' : 'Activar'} seguir puntero</button>
        </>
    )

}

    Para poder indicar la posición de la bola tendremos un estado position de tipo objecto con dos atributos, uno para la posición X y otro para la posición Y:

    const FollowMouse = () => {

    //...
    const [position, setPosition] = useState({ x: 0, y: 0 })

    return (
        <>
            <div style={{
                //...
                transform: `translate(${position.x}px, ${position.y}px)`
              }} />
            {/*...*/}
        </>
    )

}

    El estado de la posición se cambiará cada vez que el puntero se mueva, por lo que debemos suscribirnos al evento pointermove, para ello usamos el useEffect con la dependencia enabled:

    const FollowMouse = () => {

    useEffect(() => {
        console.log('effect ', { enabled })

        const handleMove = event => {
          const { clientX, clientY } = event
          console.log('handleMove', { clientX, clientY })
          setPosition({ x: clientX, y: clientY })
        }

        if (enabled) {
          window.addEventListener('pointermove', handleMove)
        }

    }, [enabled])

    // ..
}

    Como podemos comprobar si está activado veremos la bola en lugar del puntero, pero al desactivarlo seguimos viendo la bola. Esto ocurre porque nos estamos suscribiendo a un evento pero en ningún momento dejamos de suscribirnos.

    Retorno de un useEffect¶

    El useEffect puede retornar un callback, dicho método será ejecutado cuando:

    • Se desmonta el componente
    • Cada vez que se cambia una dependencia y se va a volver a ejecutar el useEffect, es decir, al cambiar la dependencia el efecto deja de producirse ejecutando el método y vuelve a ejecutarlo.

    Este return no es obligatorio, pero si es lógico usarlo cuando realizamos ciertas operaciones que queremos limpiar, como por ejemplo eliminar la suscripción a un evento, por lo que el problema del proyecto anterior se puede evitar retornando un método que elimine la suscripción al evento:

    const FollowMouse = () => {
    useEffect(() => {
        console.log('effect ', { enabled })

        const handleMove = event => {
          const { clientX, clientY } = event
          console.log('handleMove', { clientX, clientY })
          setPosition({ x: clientX, y: clientY })
        }

        if (enabled) {
          window.addEventListener('pointermove', handleMove)
        }

        return () => {
          console.log('cleanup')
          window.removeEventListener('pointermove', handleMove)
        }

    }, [enabled])

    // ...
}

    El retorno también se va a realizar cuando se desmonta el componente, es decir, cuando se elimine del DOM. Vamos a modificar nuestro proyecto, añadiendo un botón y un estado para montar y desmontar el componente FollowMouse. En caso de no tener el return en el useEffect nunca eliminaríamos la suscripción del puntero por lo que seguiríamos registrando dicho evento, a pesar de no poder ejecutar nada en el momento. Por eso, para evitar esto es conveniente usarlo

    function App() {

  const [mounted, setMounted] = useState(true)

  return (
    <main>
      {mounted && <div><FollowMouse /></div>}
      <button onClick={() => setMounted(!mounted)}>Toggle mounted FollowMouse component</button>
    </main>
  )
}

    Múltiples useEffect¶

    Se recomienda crear varios useEffect para poder realizar operaciones completamente diferentes, además de ser una buena práctica hace que el código sea más legible. En nuestro proyecto MouseEffect vamos a ocultar nuestro puntero cuando la bola esté en movimiento:

    const FollowMouse = () => {
    // ...
    useEffect(() => {
    document.body.classList.toggle('no-cursor', enabled)

    return () => {
        document.body.classList.remove('no-cursor')
    }
    }, [enabled])

    //...
}

    Para eso creamos un nuevo efecto que dependa del estado enabled, es decir, si está activado o no el seguir al puntero, vamos a indicar una clase al body de nuestra página (no-cursor). Como función de retorno, devolveremos un método para que elimine esa clase del body, ya que el body es una etiqueta global y si desmontamos nuestro componente seguirá existiendo.

    React.StrictMode¶

    Cuando estamos desarrollando un proyecto React decimos que estamos en modo desarrollo. Cuando construimos un proyecto, ya sea con el comando create-react-app o con vite, nuestro componente está en vuelto en un componente de React, StrictMode:

    ReactDOM.createRoot(document.getElementById('root')).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>,
)

    React.StrictMode es un componente especial de React que se utiliza para ayudar a encontrar y corregir problemas en la aplicación relacionados con el rendimiento y las prácticas no seguras en el código en modo desarrollo. Puedes envolver partes de tu aplicación con React.StrictMode para activar un conjunto de comprobaciones adicionales de tiempo de desarrollo que te alertarán sobre posibles problemas.

    Al utilizar React.StrictMode, se habilitan las siguientes comprobaciones:

    • Detección de efectos secundarios no deseados: React.StrictMode ayuda a identificar efectos secundarios no intencionales en tus funciones de renderizado. Por ejemplo, si una función de renderizado tiene efectos secundarios y se llama más de una vez, StrictMode emitirá una advertencia. De ahí que, en modo desarrollo, nuestro useEffect se ejecute dos veces al inicializar el componente.
    • Advertencias sobre el uso de funciones de ciclo de vida obsoletas: Si tu código utiliza funciones de ciclo de vida de componentes de clase que están obsoletas o se han eliminado en versiones futuras de React, StrictMode te dará advertencias sobre su uso.
    • Detección de renderizados redundantes: StrictMode puede detectar renderizados redundantes en tus componentes, lo que puede ayudarte a optimizar el rendimiento.
    • Advertencias sobre el uso de FindDOMNode: Si tu aplicación utiliza findDOMNode para acceder al DOM directamente, StrictMode emitirá advertencias. En lugar de usar findDOMNode, se recomienda utilizar ref para acceder a los nodos del DOM.

    Es importante destacar que las comprobaciones realizadas por React.StrictMode solo se aplican en el entorno de desarrollo y no afectarán a la producción. Por lo tanto, es seguro utilizar React.StrictMode durante el desarrollo para identificar y solucionar problemas potenciales, y luego puedes retirarlo en el entorno de producción para evitar el impacto en el rendimiento.

    Cuando desplegamos nuestra aplicación y entramos en modo producción, el React.StrictMode no entra en juego, por lo que esas comprobaciones quedarán deshabilitadas, y por ejemplo, el useEffect solo se realizará una vez al inicializar el componente.

    Problemas useEffect¶

    Uno de los problemas de useEffect es que si no se realiza correctamente puede provocar un bucle infinito. Esto además de ser peligroso y de ejecutar infinitamente una función, es una mala práctica.

    El useEffect se puede ejecutar de forma infinita:

    • Cuando no se indica dependencia (ni una lista vacía) y se ejecuta un cambio de estado:

      `javascript useEffect(() => { setState(10) })

    • Cuando se indica una dependencia y se hace cambio del estado de dicha dependencia:

      ```javascript useEffect(() => { setState(10) }, [state])

    Otro de los problemas de useEffect es que no puede recibir parámetros en su callback, por lo que si se desea pasarle alguna variable, debemos crear la variable fuera y acceder a ella dentro del useEffect:

    const value = 10

useEffect(() => {
    console.log(value)
}, [])
