    2 Oyentes y cadenas¶

    Proyecto CountingClicks¶

    En este proyecto desarrollaremos una aplicación que muestra el pantalla el número de veces que se ha hecho click sobre un botón.

    Crea un proyecto mediante el asistente, tal y como vimos anteriormente. Después modifica la especificación XML de layout de la actividad, de manera que el archivo pase a llamarse res/layout/main_activity.xml y que contenga:

    <LinearLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:tools="http://schemas.android.com/tools"
    android:id="@+id/llRoot"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:orientation="vertical"
    android:padding="@dimen/main_margin">

    <TextView
        android:id="@+id/lblTimes"
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        tools:text="0"
        android:textAppearance="@style/TextAppearance.AppCompat.Display3"/>

    <Button
        android:id="@+id/btnClickMe"
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:layout_marginTop="@dimen/main_margin"
        android:text="@string/main_btnClickMe"/>

</LinearLayout>

    Algo de lo que no hemos hablado aún es que el contenedor LinearLayout se caracteriza porque apila sus vistas hijas de forma horizontal (de izquierda a derecha) o de forma vertical (de arriba a abajo) dependiendo del valor de su atributo android:orientation, como estudiaremos con más detenimiento más adelante. Así que en nuestro diseño el botón aparecerá debajo del TextView.

    Como vemos, en el layout anterior hemos establecido que el ViewGroup raíz tenga un relleno (padding) para que los elementos no se peguen al borde. Además, se le ha aplicado un estilo predefinido al TextView que va a mostrar el número de veces que se ha hecho click, de manera que se vea bastante grande. Finalmente, se ha establecido un margen superior en el botón para que éste no quede pegado justo debajo del TextView.

    Un aspecto curioso es que no se ha establecido texto al TextView, ya que el texto inicial se establecerá en tiempo de ejecución desde código Java o Kotlin, e irá cambiando conforme el usuario haga click en el botón. Sin embargo hemos especificado mediante el atributo tools:text que el editor visual muestre el un texto para poder comprobar el diseño correctamente.

    Hemos definido los siguientes recursos de valor en el archivo res/values/strings.xml:

    <resources>
    <string name="app_name">CountingClicks</string>
    <string name="main_btnClickMe">Click me!</string>
</resources>

    y en el archivo res/values/dimens.xml:

    <resources>
    <dimen name="main_margin">16dp</dimen>
</resources>

    Listeners¶

    En la programación de aplicaciones con interfaces de usuario gráficas es muy usual que las acciones realizadas por el usuarios, conocidos como eventos, sean notificados al código de programación a través de lo que se conoce como callbacks o listeners.

    Un listener no es más que una interfaz que contiene uno o más métodos de callback que serán ejecutados por el sistema cuando se produzca el evento correspondiente en una determinada vista.

    Nuestra misión como programadores será crear un objeto de una clase que implemente dicha interfaz listener para que el o los métodos correspondientes contengan el código de respuesta al evento producido por el usuario. Por ejemplo, la interfaz View.OnClickListener define un método de callback llamado onClick(view) que será llamado por el sistema cuando el usuario haga click sobre una vista, que será pasada como argumento al método.

    Una vez creado dicho objeto listener, deberemos asociarlo a la vista correspondiente, para que el sistema sepa sobre qué objeto debe ejecutar el método de callback cuando el usuario produzca el evento en la vista. Por ejemplo, llamaremos al método setOnClickListener(onClickListener) de una determinada vista para establecer el listener OnClickListener a cuyo método onClick(view) queremos que el sistema llame cuando el usuario haga click sobre dicha vista.

    Como podemos intuir, los listener son los candidatos perfectos para usar la sintaxis Java de clases inline anónimas, ya que debemos crear un objeto específico para la ocasión y en realidad nos da igual cómo se llame la clase porque sólo vamos a crear un único objeto de ella.

    Por ejemplo, podemos crear un un objeto de una clase anónima que implementa la interfaz OnClickListener de manera inline, justo cuando se está estableciendo como listener para una determinada vista a través de su método setOnClickListener(onClickListener). El código de nuestra actividad quedaría:

    public class MainActivity extends Activity {

    private MainActivityBinding binding;
    private int times;

    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        binding = MainActivityBinding.inflate(getLayoutInflater());
        setContentView(binding.getRoot());
        setupViews();
    }

    private void setupViews() {
        binding.btnClickMe.setOnClickListener(new OnClickListener() {
            public void onClick(View v) {
                incrementAndShow();
            }
        });
    }

    private void incrementAndShow() {
        times++;
        showTimes();
    }

    private void showTimes() {
        binding.lblTimes.setText(String.valueOf(times));
    }

}

    Como desventaja podríamos indicar que utiliza una sintaxis algo compleja y que el código queda poco limpio, y no permite recibir datos adicionales, ya que en la sintaxis de clase anónima inline no podemos definir el constructor de la clase.

    Las expresiones lambda de Java 8 nos proporcionan una sintaxis más sencilla que permite aumentar la legibilidad de nuestro código en relación al empleo de clases inline anónimas que implementan una determinada interfaz funcional, que usamos habitualmente al especificar los listener de nuestras vistas.

    Interfaz funcional

    Una interfaz funcional es aquella que dispone de un único método abstracto, de ahí que también reciban el nombre de SAM (single abstract method, único método abstracto).

    En el caso de un listener contenga más de un método abstracto, no podremos usar expresiones lambda, y deberemos seguir usando clases anónimas inline.

    Por ejemplo, si queremos establecer el listener para el código anterior haciendo uso de una expresión lambda haríamos:

    public class MainActivity extends Activity {

    // ...

    private void setupViews() {
        binding.btnClickMe.setOnClickListener(v -> incrementAndShow());       
    }

}

    Una expresión lambda tiene la sintaxis (param1, param2) -> { cuerpo }, con las siguientes características:

    • Podemos omitir el tipo de dato de cada parámetro, ya que el compilador puede deducirlos.
    • Si se especifica un único parámetro, podemos omitir los paréntesis. Si no se especifica ningún parámetro es obligatorio poner los paréntesis.
    • En el cuerpo podemos omitir las llaves si contiene una única expresión o una única sentencia que no retorna valor.
    • Si el cuerpo contiene una única expresión, ésta será evaluada y la expresión lambda retornará el valor obtenido. Ojo, debemos tener en cuenta que return no es una expresión, por lo que si es la única sentencia del cuerpo deberemos usar llaves.
    • Si el cuerpo contiene una única sentencia return valor, forzosamente debemos poner las llaves, ya que return no es una expresión. En este caso la expresión lambda retornará el valor retornado por return.

    Sugerencia

    Si el cuerpo de nuestra lambda contiene varias sentencias, se recomienda extraer dichas sentencias hacia un método y hacer que la lambda contenga una única instrucción correspondiente a la llamada a dicho método. De esta manera se incrementa notablemente la legibilidad de nuestro código, ya que no será necesario usar llaves.

    En el caso de que el cuerpo de la expresión lambda sea una llamada a un método que recibe los mismos parámetros de la expresión lambda, podemos usar incluso una sintaxis aún más simple, a través de una method reference (referencia a un método). Las method reference se expresan con la siguiente sintaxis objeto::nombreMetodo o clase::nombreMetodo (si se trata de un método estático, o de un objeto no expresado explícitamente).

    Así, el ejemplo anterior podría ser modificado de la siguiente forma para usar method reference:

    public class MainActivity extends Activity {

    // ...

    private void setupViews() {
        binding.btnClickMe.setOnClickListener(this::incrementAndShow);
    }

    private void incrementAndShow(View v) {
        times++;
        showTimes();
    }

}

    Warning

    Para poder usar las expresiones lambda o las method references en nuestra aplicación debemos hacer que nuestro proyecto use la versión JDK 8 de Java.

    Para asegurarnos de que nuestro módulo trabaja con la versión 8 de Java debemos comprobar que el archivo build.gradle de nuestro módulo app contiene la siguiente configuración:

    android {
    // ...
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_1_8
        targetCompatibility = JavaVersion.VERSION_1_8
    }
}

    Warning

    Debemos tener en cuenta que si la API mínima de nuestro proyecto no es la versión 24, no podremos usar todas las funcionalidades de Java 8, incluso aunque lo hayamos indicado en el archivo build.gradle de nuestro módulo. Para saber más sobre las funcionalidad de Java 8 disponibles dependiendo de la API mínima especificada visite la web oficial de desarrollo Android

    Si tratamos de usar expresiones lambda si haber añadido las líneas anteriores en el archivo build.gradle Android Studio nos mostrará un error y nos indicará si queremos que añada dichas líneas automáticamente.

    Recursos de cadenas con parámetros¶

    Suele ser muy habitual que tengamos que introducir valores calculados por nuestro programa dentro de un mensaje que se le quiera mostrar al usuario. Por ejemplo, supongamos que queremos mostrar el mensaje 3 times, donde 3 corresponde al valor de una determinada variable.

    La primera posibilidad es simplemente construir el mensaje total concatenando sus distintas partes, esto es

    public class MainActivity extends Activity {

    // ...

    private void showTimes() {
        binding.lblTimes.setText(times + " " + getString(R.string.main_times));
    }

}

    habiendo definido en el archivo res/values/strings.xml el recurso de cadena

    <string name="main_times">times</string>

    Afortunadamente, Android nos proporciona un método más adecuado para tratar este tipo de situaciones. Así, podemos especificar que un determinado recurso de cadena contenga una serie de parámetros, de manera que al obtener el recurso podamos indicar los argumentos correspondientes a los valores de los parámetros.

    Sugerencia

    Siempre es preferible que definas recursos de cadena con parámetros frente a concatenar valores y cadenas, incluso en el caso de Kotlin, y sus template string (que estudiaremos más tarde), ya que la definición de recursos de cadena nos ayuda a la internacionalización de nuestra aplicación, de manera que la cadena se pueda mostrar en el idioma del usuario.

    Estos recursos siguen siendo recursos de cadena normales, por lo que normalmente se definen en el archivo /res/values/strings.xml. Así en el ejemplo anterior crearíamos el recurso de cadena:

    <string name="main_times">%1$d times</string>

    donde el 1$indica que el valor asignado a dicho parámetro corresponderá al valor proporcionado como primer argumento adicional y el %d indica que el valor del argumento deberá ser un valor entero (si fuera una cadena sería %s y si fuera un valor real %f). Si tuviéramos más de un parámetro es segundo sería 2$, el tercero 3$ y así sucesivamente.

    Sugerencia

    El uso de 1$, 2$ no es obligatorio, es decir, podemos indicar simplemente %d, %s, etc., pero se recomienda especificar el número de argumento por si en un idioma diferente tenemos que cambiar el orden en el que usan los argumentos en la cadena de formato.

    Para poder especificar los argumentos correspondientes a los valores que deben tomar los parámetros, a la hora de obtener el recurso usaremos el método getString(resId, arg…) de la clase Resources, que está sobrecargado para recibir, además del resId de la cadena, los valores que queremos que tomen los parámetros (que deberemos pasar en el orden adecuado). Esta versión de getString() también existe directamente en nuestra actividad, sin tener ni siquiera que obtener el objeto Resources. Por ejemplo:

    public class MainActivity extends Activity {

    // ...

    private void showTimes() {
        binding.lblTimes.setText(getString(R.string.main_times, times));
    }

}

    Ten en cuenta

    Internamente el método getString() usa el método format de la clase String de Java, y por este motivo la especificación de los valores es exactamente el mismo que este último.

    Plurals¶

    En algunas ocasiones el texto de un recurso de cadena depende de una cierta cantidad numérica. Supongamos que queremos mostrar un texto con el número de veces que se ha hecho click; en este caso el texto podría ser "1 time", "2 time**s**". Como vemos, dependiendo de si es más de una vez debemos usar el plural en vez del singular.

    Aunque podríamos definir dos recursos de cadena distintos y decidir cuál usar dependiendo del valor de la variable, Android nos proporciona una funcionalidad muy interesante consistente en definir un único recurso de cadena que contenga en su interior la especificación de las distintas cadenas que deben usarse dependiendo de una determinada cantidad. Estos recursos reciben el nombre de recursos de plural (plurals). Veamos un ejemplo:

    <plurals name="main_times">
    <item quantity="one">%1$d time</item>
    <item quantity="other">%1$d times</item>
</plurals>

    Sugerencia

    Los recursos de plural suelen definirse dentro de su propio fichero /res/values/plurals.xml, aunque al ser recursos de valor no es estrictamente necesario, por lo que podríamos definirlos, por ejemplo, dentro del archivo string.xml.

    El atributo quantity puede tomar uno de los siguientes valores: zero (cero), one (uno), two (dos), few (pocos, tres o cuatro), many (muchos, más de 10), other (resto de casos).

    Warning

    Algunos de estos valores pueden no tener efecto dependiendo del idioma del que se trate. Por ejemplo, en el idioma español, sintácticamente hablando, el uso del singular/plural para la cantidad cero no difiere del uso del singular/plural de cualquier valor distinto de 1, como por ejemplo "0 veces" y "4 veces". Por este motivo si usamos los valores zero y other a la vez en la definición de un plural y el idioma del dispositivo corresponde al español, la cadena asignada a zero nunca es usada, aunque el valor de la variable sea 0, sino que en su lugar se usa la cadena asignada a other. En español sucede exactamente igual con los valores two, few o many.

    Para poder obtener un recurso de plurales desde nuestro código Java o Kotlin debemos usar el método getQuantityString(resId, quantity) de la clase Resources, al que le pasaremos el resId del plural y la cantidad a usar para determinar cuál de las cadenas debe mostrarse.

    Ten en cuenta

    A diferencia de getString(), el método getQuantityString() no se encuentra definido en la clase Activity, sino tan sólo en la clase Resources, por lo que tendremos forzosamente que obtener desde objeto desde la actividad.

    Si además las cadenas incluidas dentro del recurso plural contienen parámetros, entonces podemos usar una versión sobrecargada del método getQuantityString(resId, quantity, arg…), que recibe el resId del plural, la cantidad a usar para determinar cuál de las cadenas debe usarse, y además otra serie de argumentos correspondientes a los valores para los parámetros de la cadena elegida.

    Así, en nuestro proyecto haríamos:

    public class MainActivity extends Activity {

    // ...

    private void showTimes() {
        binding.lblTimes.setText(
            getResources().getQuantityString(R.plurals.main_times, times, times));
    }

}

    en cuyo caso, el primer times es usado para determinar qué cadena escoger de entre las pertenecientes al recurso de plural, y el segundo times se usa como argumento del parámetro de la cadena seleccionada.

    Ten en cuenta

    En los sistemas operativos móviles no existe el concepto de click con botón derecho del ratón, ya que normalmente no se usa ningún dispositivo apuntador, sino que la pantalla del dispositivo es táctil. Sin embargo existe una acción similar conocida como hacer click largo, consistente en dejar pulsada la vista durante más tiempo del habitual.

    Para saber más sobre esta acción investiga en la documentación oficial https://developer.android.com/index.html acerca de la interfaz OnLongClickListener.
