    4 Recursos y la Clase R¶

    Introducción¶

    El diseño de una aplicación Android incorpora muchos elementos que no cambian a lo largo de su vida, como imágenes, constantes de cadena, etc. Estos elementos reciben el nombre de recursos. Cuando desarrollamos aplicaciones en Android debemos siempre tratar de mantener la separación entre, por un lado, el código de la aplicación, y por otro lado, los recursos que utiliza, porque esto nos va a permitir proporcionar recursos alternativos para las distintas configuraciones de dispositivos, como por ejemplo diferentes idiomas o tamaños de pantalla, lo que es especialmente relevante dada la variedad de dispositivos Android existentes en el mercado.

    Los recursos de un proyecto deben almacenarse en la carpeta src/main/res/ del módulo correspondiente, y serán empaquetados en el fichero .apk cuando generemos el instalable de nuestra aplicación. Como vimos cuando hablamos de la estructura de un proyecto Android, los ficheros de recursos deben estar organizados en una serie de subcarpetas atendiendo al tipo de recurso del que se trate. Por ejemplo: en la carpeta src/main/res/layout/ se colocarán los recursos por defecto correspondientes a especificaciones XML de layouts, en src/main/res/menu/ los recursos por defecto correspondientes a especificaciones XML de menús, etc.

    Para referenciar a estos recursos desde código XML utilizaremos la siguiente sintaxis:

    @[paquete:]tipo_recurso/nombre_recurso_sin_extensión. Por ejemplo:

    • Archivo de layout activity_main.xml almacenado en la carpeta res/layout: usaremos @layout/activity_main.
    • Identificador txtHello: usaremos @id/txtHello o @+id/txtHello si queremos que se cree el id si no existe.
    • Recurso de constante de cadena de caracteres con nombre hello: usaremos @string/hello
    • Archivo de imagen ic_launcher.png almacenada en la carpeta res/drawable: usaremos @drawable/ic_launcher

    El empleo del nombre de paquete es opcional, tanto en el acceso desde código Java como desde código XML, ya que no es necesario usarlo cuando el recurso se encuentre en el mismo paquete en el que se encuentre el código desde el que se quiere acceder.

    Si, por el contrario, lo que queremos es referenciar dichos recursos desde código Java o Kotlin, el entorno de desarrollo y el framework nos facilita mucho la tarea. Así, el sistema identificará cada recurso con un valor entero conocido como resId que podremos usar para hacer referencia a él.

    resId

    Un resId (resource id, identificador de recurso) es un valor entero que el sistema asigna automáticamente a un recurso para que podamos referenciarlo tanto desde código XML como desde código Java o Kotlin.

    De hecho, muchos métodos del framework que reciben recursos como parámetro están diseñados para que reciban su resId. Por ejemplo, el método setContentView(layoutResId) de la clase AppCompatActivity está diseñado para que reciba un entero correspondiente al resId del recurso de especificación XML de layout que se quiere establecer como interfaz de usuario de la actividad.

    Sin embargo, ello nos obligaría a recordar qué valor entero corresponde a cada recurso, lo que realmente parece imposible. Para solucionar este problema, cada vez que se compila nuestra aplicación o cada vez que añadimos, modificamos o eliminar un recurso, se autogenerará una clase Java llamada R.

    La clase R

    La clase R es una clase Java generada automáticamente por el entorno de desarrollo cada vez que agregamos, modificamos o eliminamos un recuso de nuestro proyecto. Dicha clase contendrá una constante para cada recurso de nuestro proyecto, cuyo nombre corresponderá al nombre del recurso y cuyo valor corresponderá al resId que le ha asignado el sistema.

    En concreto, la clase R contendrá una subclase estática para cada tipo de recurso existente (por ejemplo R.drawable, R.layout, R.string), que, a su vez, contendrá una constante entera estática cuyo nombre corresponderá al nombre del recurso y su valor será el resId que el sistema le haya asignado automáticamente.

    El nombre del recurso depende del tipo de recurso del que se trate. Si se trata de un recurso representado por un archivo, como por ejemplo una imagen icono.png el nombre del recuso corresponderá al nombre del archivo sin extensión, en este caso icono. Para otros tipos de recursos el nombre del recurso se indica explícitamente en su declaración, como en el caso de los recursos de constantes de cadena que veremos más adelante.

    public final class R {

    public static final class drawable {
        public static final int ic_launcher=0x7f020000;
    }
    public static final class id {
        public static final int textView1=0x7f050000;
    }
    public static final class layout {
        public static final int activity_main=0x7f030000;
    }
    public static final class string {
        public static final int app_name=0x7f040001;
        public static final int hello=0x7f040000;
    }

    // ...

}

    Gracias a esta clase autogenerada de constantes, podemos referenciar fácilmente el resId asociado a un determinado recurso desde código Java o Kotlin. Bastará con usar la sintaxis [paquete.]R.tipo_recurso.nombre_recurso, ya que simplemente estaremos accediendo a una constante Java (public static final) definida dentro de una subclase estática de la clase R. Por ejemplo:

    • Archivo de layout activity_main.xml almacenado en la carpeta res/layout: usaremos R.layout.activity_main
    • Identificador txtHello: usaremos R.id.txtHello
    • Recurso de constante de cadena de caracteres con nombre hello: usaremos R.string.hello
    • Archivo de imagen ic_launcher.png almacenada en la carpeta res/drawable: usaremos R.drawable.ic_launcher

    La clase R será generada en el archivo /app/build/generated/not_namespaced_r_class_sources/debug/processDebugResources/r/<paquete>/R.java.

    Danger

    Dado que la clase R es generada automáticamente en el proceso de compilación, nunca debemos editarla manualmente.

    Warning

    En las versiones más recientes del plugin de Gradle para Android Studio la clase R será generada directamente en bytecode, sin crear un fichero R.java intermedio. Si quisiéramos ver su contenido tendríamos que descompilar el bytecode correspondiente.

    Recursos ID¶

    En Android, los id con los que vamos a identificar las vistas son considerados recursos, ya que su valor no cambia a lo largo del tiempo.

    Sin embargo, y a diferencia de lo que sucede con el resto de tipos de recursos, para los id no es necesario crear ningún fichero ni subcarpeta especial dentro de la carpeta app/src/main/res/, ni añadir ninguna declaración en un archivo de recursos. Simplemente, cuando en una especificación de layout XML en la que queramos referenciar a un nuevo recurso de id deberemos usar la sintaxis

    <TextView
    android:id="@+id/txtHello"
    ... />

    El carácter + le indica al entorno de desarrollo que debe generar automáticamente un nuevo recurso de id con el nombre txtHello, siempre y cuando dicho recurso no exista ya. Posteriormente, se obtiene el valor entero generado automáticamente para dicho recurso (su resId) y se establece como valor de la propiedad id de la vista correspondiente.

    Como ya sabemos, cuando se genere la clase Java llamada R, se creará automáticamente una subclase estática, llamada id, que contendrá una constante entera estática para cada uno de los id que hayamos creado.

    public final class R {

    public static final class id {
        public static final int llRoot=0x7f07008c;
        public static final int txtHello=0x7f07008d;
    }

    // ...

}

    Esto nos permitirá acceder desde código Java o Kotlin al resId del recurso de id.

    Warning

    Un recurso de id NO es lo mismo que un resId. Un recurso de id es un tipo de recurso, similar a un recurso de cadena o un recurso de layout, mientras que el resId es el número entero que el sistema asigna como valor a cada una de las constantes definidas en la clase R que representan recursos.

    Referencia a vistas de código Java o Kotlin¶

    Como hemos comentado, una de las tareas que llevamos a cabo durante la creación de la actividad es generar el árbol de vistas a partir de la especificación XML del layout de la interfaz de usuario. Como resultado del proceso de inflado realizado por setContentView(layoutResId) se obtiene el elemento raíz de dicho árbol, que es establecido como interfaz de usuario de la actividad.

    Sin embargo, habitualmente necesitaremos obtener una referencia a vistas concretas dentro del árbol de vistas, para por ejemplo establecer alguna de sus propiedades o detectar la interacciones del usuario con ellas.

    Para obtener la referencia a una vista concreta deberemos realizar una operación de búsqueda dentro del árbol correspondiente a partir de su vista raíz. Para poder llevar a cabo dicha operación, la clase AppCompatActivity proporciona el método findViewById(viewIdResId), que recibe el resId del recurso de id asignado a la vista que queremos buscar dentro del árbol de vistas correspondiente a la interfaz de usuario de la actividad. Como resultado obtendremos un objeto de la clase correspondiente (que deberá heredar de View), dependiendo del tipo de la variable a la que asignemos la referencia.

    De hecho, será muy habitual que tengamos que obtener la referencia a varias vistas de la interfaz de usuario, por lo que se suele crear un método para tal fin, llamado por ejemplo setupViews(), en el que se lleven a cabo las operaciones de búsqueda y de configuración de las vistas. El proceso de búsqueda de una vista dentro del árbol puede resultar bastante costoso en tiempo, por lo que se recomienda realizarlo un única vez, así que llamaremos al método setupViews() durante la creación de la actividad, normalmente en el método onCreate().

    Como es bastante probable que debamos acceder a los objetos que representan a las vistas desde distintos métodos de nuestra clase actividad, es habitual definir la variables que referencian vistas como fields (campos) de la clase actividad.

    public class MainActivity extends AppCompatActivity {

    private TextView txtHello;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
        setupViews();
    }

    private void setupViews() {
        txtHello = findViewById(R.id.txtHello);
        txtHello.setText("Hello. What a wonderful world!");
    }

}

    Nunca hagas esto

    No podemos realizar el proceso de búsqueda en la propia declaración del field correspondiente a la vista, dado que en ese momento a la actividad aún no se le ha establecido su interfaz de usuario.

    public class MainActivity extends AppCompatActivity {

    // ¡¡¡ NUNCA HAGAS ESTO. DA ERROR DE EJECUCIÓN NullPointerException !!!
    private TextView txtHello = findViewById(R.id.txtHello);

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
    }

}

    Otro aspecto que debemos tener en cuenta es qué ocurre si la vista cuyo id es proporcionado al método findViewById(viewIdResId) no existe en el layout de la actividad, por ejemplo porque se trate de un id válido de una vista de la interfaz de usuario de otra actividad.

    Warning

    Si la vista cuyo id es proporcionado al método findViewById(viewIdResId) no existe en el layout de la actividad el método findViewById() retornará null, ya que no podrá encontrar la vista en el árbol de vistas correspondiente a la interfaz de usuario de la actividad. Esto puede resultar peligroso si no lo controlamos, dado que más adelante cuando queramos acceder a una propiedad de dicha vista se producirá una excepción NullPointerException.

    Para solventar este inconveniente, la librería de soporte proporciona el método estático ActivityCompat.requireViewById(activity, viewIdResId), que a diferencia de findViewById(viewIdResId) no retornará nunca null, sino que lanzará la excepción IllegalArgumentException si no existe una vista con dicho id en el layout de la actividad.

    Esto tiene dos ventajas: la variable destinataria no puede recibir null como consecuencia de la llamada y la excepción se lanza justo en la línea que es errónea, lo que nos permite depurar el error más rápidamente.

    public class MainActivity extends AppCompatActivity {

    private TextView txtHello;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
        setupViews();
    }

    private void setupViews() {
        txtHello = ActivityCompat.requireViewById(this, R.id.txtHello);
        txtHello.setText("Hello. What a wonderful world!");
    }

}

    Ten en cuenta

    Como veremos cuando empecemos a estudiar Kotlin, el empleo de ActivityCompat.requireViewById(activity, viewIdResId) no tiene sentido ya que si definimos la variable como de un tipo no nullable, si findViewById(viewIdResId) retorna null se producirá un error en la ejecución, ya que la variable no puede tomar el valor null.

    Por otra parte, debemos tener en cuenta que se puede producir un error de conversión por tipo incompatible con respecto al tipo de la variable destinataria.

    Warning

    Tanto al hacer findViewById() como ActivityCompat.requireViewById() se puede producir un error de conversión de tipo si el tipo especificado para la variable no es compatible con el tipo de la vista del layout que posee el id pasado como argumento.

    Por ejemplo, si en el layout que estamos viendo hacemos

    // Da ERROR en tiempo de ejecución
LinearLayout txtHello = ActivityCompat.requireViewById(this, R.id.txtHello);

    Se producirá un error en tiempo de ejecución, ya que la vista con id txtHello corresponde a un TextView, que ni es ni hereda de LinearLayout.

    Desde el punto de vista del desarrollo, la codificación de todas estas llamadas a findViewById() o ActivityCompat.requireViewById() es un proceso bastante tedioso, por lo que a lo largo del tiempo han ido apareciendo distintas opciones para automatizarlo. La mejor de ellas es la proporcionada por la funcionalidad conocida como ViewBinding.
